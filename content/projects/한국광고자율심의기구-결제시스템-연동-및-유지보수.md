## 프로젝트 개요

광고 심의 신청·심사 업무를 온라인으로 처리하는 운영 중인 웹 서비스로, 회원(광고주·대행사)이 심의신청서를 작성해 수수료를 결제하면 관리자가 심의·의결하고 심의필증을 발급하는 시스템입니다.

제가 처음부터 설계한 프로젝트가 아니라, 이미 운영 중이던 서비스에 '소방수' 성격의 유지보수 인력으로 갑작스럽게 투입되어, 기존 코드를 빠르게 파악하며 전반에 산재한 오류를 대응하는 것이 시작점이었습니다. 초기에는 서비스 곳곳에서 발생하던 버그·에러를 폭넓게 수정하는 데 집중했고, 이후에는 매출과 직결되는 '심의신청 → 결제 → 승인 → 조회' 핵심 플로우를 집중적으로 개발·개선했습니다.

## 담당 업무

- **폭넓은 유지보수 대응**: 운영 중인 서비스에 투입되어 회원(가입·본인인증·마이페이지), 심의신청, 결제, 게시판/자료실, 관리자 백오피스 등 여러 도메인에 걸쳐 발생하던 오류를 파악하고 수정했습니다.
- **심의신청서 기능**: 광고 카테고리(방송·인쇄·숏폼·인터넷/기타 등)별 신청 폼에서 회사·사업자, 광고 정보, 신청자 정보 입력과 파일 임시 업로드를 비동기로 처리하고, 제출 시 관련 데이터를 여러 테이블에 나누어 저장하도록 구성했습니다. 신청번호는 대행 구분·신청 구분·카테고리·연도와 순번을 조합한 규칙으로 자동 채번되도록 했습니다.
- **KG이니시스 결제 연동**: PG사는 KG이니시스 PC 표준결제를 사용했으며, 결제 요청 시 서명·검증값을 생성하고 승인 단계에서 인증 정보를 대조한 뒤 승인 API를 호출해 위·변조를 방지했습니다. 단건 결제뿐 아니라 여러 신청서를 한 번에 결제하는 다건 결제도 지원하도록 구현했습니다.
- **결제 데이터 정합성 처리**: PG 결제창으로 이동했다가 돌아오는 과정에서 세션이 끊길 수 있는 문제를, 임시 결제 토큰을 발급·저장해 복귀 시 사용자·신청 정보를 복원하는 방식(일정 시간 후 만료, 사용 즉시 삭제)으로 해결했습니다. 승인 API가 실패하면 망취소를 자동 호출해 미승인 결제가 남지 않도록 처리하고, 실패 시 안내 페이지로 이동시켰습니다.
- **신청완료·승인 처리**: 결제 성공 시 거래 내역을 저장하고 신청 건의 결제 상태를 갱신하도록 구현했습니다. 관리자는 백오피스에서 심의 결과(승인/반려, 반려 시 사유)를 저장하고, 심의필증을 발급하면서 심의번호를 채번하도록 했습니다. 필증 발급 시 신청자에게 이메일을 자동 발송하고 이력을 기록하도록 구현했습니다.
- **결제 취소·환불 처리(관리자)**: 관리자 화면에서 이니시스 환불 API를 호출해 결제를 취소하고, 성공 시 결제 상태와 신청 상태를 취소 처리로 갱신하도록 구현했습니다.
- **신청서·심의필증 조회**: 사용자 마이페이지에서 심의대기·반려·완료 상태별로 탭을 나누고, 대기 탭 안에서도 미결제·결제완료·취소 상태를 다시 필터링해 비동기로 조회하도록 구현했습니다. 결제완료 건은 주문 단위로 묶어서 표시하고, 심의필증은 서버가 아닌 클라이언트 단에서 PDF로 생성해 다운로드·인쇄할 수 있도록 구현했습니다.

## 개선 과정

- **기존 코드 파악의 어려움**: 문서와 이력이 정리되지 않은 운영 코드에 급하게 투입되어, 절차적 구조로 얽힌 신청·결제·심의 로직을 코드를 직접 읽어가며 역추적하고 수정 범위를 좁혀 나가야 했습니다. 정리되지 않은 과거 버전 코드가 함께 남아 있는 경우도 있어, 실제 운영 중인 로직과 구분해가며 작업했습니다.
- **결제 정합성 확보**: 결제 작업에서 가장 신경 쓴 부분은 "결제는 완료됐는데 신청 상태는 갱신되지 않는" 불일치를 막는 것이었습니다. PG 리다이렉트 구간의 세션 유실 문제는 결제 토큰 기반 컨텍스트 복원으로, 승인 실패·예외 상황은 망취소 자동 호출로, 승인 응답 위·변조 가능성은 서명·인증 정보 대조로 각각 방어하도록 구성했습니다.
- **주문 단위 관리**: 여러 심의신청서를 한 번에 결제하는 경우를 하나의 주문으로 묶어, 조회 시 주문 단위로 그룹핑해 보여주도록 정리했습니다.
- **상태 모델 정리**: 결제 상태(미결제/결제완료/취소)와 심의 결과 상태(대기/승인/반려/필증발급)라는 두 축으로 신청 건의 상태를 일관되게 판별하도록 조회·표시 로직을 맞췄습니다.
